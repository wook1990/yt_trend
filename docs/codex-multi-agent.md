# Codex Multi-Agent Harness

This file is generated by `./scripts/sync-agents.sh`.
It explains how shared agent definitions should be used when coordinating work in Codex.

## Goals

- Keep a single shared prompt source in `/Users/haram/workspace/agents`.
- Preserve folder structure so specialized agents can be organized by domain.
- Route planning, implementation, and review work predictably.

## Coordination Model

1. Pick the narrowest agent whose source path matches the task domain.
2. If no domain-specific agent exists, fall back to the root generic agents.
3. For multi-phase work, apply agents in this order: planner -> implementer -> reviewer.
4. Keep ownership boundaries clear when parallel work is used.
5. Do not let one agent redefine repository-wide rules already stated in `AGENTS.md`.

## Suggested Folder Convention

- `workspace/agents/planner.md`: generic planning role
- `workspace/agents/reviewer.md`: generic review role
- `workspace/agents/implementer.md`: generic implementation role

- `workspace/agents/backend-data/*.md`: shared contracts and mock-data specialists

- `workspace/agents/desktop/*.md`: desktop client specialists
- `workspace/agents/data-science/*.md`: analysis, model, and experiment specialists
- `workspace/agents/product/*.md`: product flow planning specialists
